# 15 — FAQ

## Q1. Is this legal?
**A.** Yes, when run inside the authorized air-gapped lab described in this
documentation. The honeypots and the attacker all run on systems we own, on
a private network with no Internet uplink. Unauthorized use against
third-party systems is a criminal offense.

## Q2. Why is the model NOT deployed?
**A.** Three reasons (see `docs/10_Model_Training.md` §10.11 and `docs/14_Future_Work.md`):
1. The academic objective is the *methodology* (prove honeypot logs can
   train a classifier), not a production system.
2. Deploying an inline classifier would change the honeypot's behavior
   (blocking attacks defeats the data-collection purpose).
3. Deployment is a separate engineering project with its own latency,
   safety, and rollback requirements, documented in `trained_models/README.md`
   and `docs/14_Future_Work.md`.

## Q3. Could I just deploy the saved model in my company's SIEM?
**A.** Yes. The `trained_models/` folder contains everything needed: the
joblib model, the scaler, the feature column order, and the label mapping.
The `README.md` in that folder shows a minimal inference example. In a real
deployment you would:
- Wrap the model in a small HTTP microservice.
- Have your SIEM call it for each candidate event.
- Alert on high-probability malicious predictions.

## Q4. Why Docker and not VMs for the honeypot?
**A.** Docker gives process-level isolation with millisecond spin-up,
per-container stdout logging that Filebeat can autodiscover, and a tiny
resource footprint. VMs would give stronger isolation but are heavier and
harder to log centrally without an agent per VM.

## Q5. Why ELK and not Splunk?
**A.** ELK is open-source (free for the lab), has the richest Kibana
visualization ecosystem, and is the de-facto SIEM in many SOCs. Splunk
would be a valid alternative but its free tier is limited to 500 MB/day.

## Q6. Why 11 containers and not 50?
**A.** 11 covers the major IoT/server classes (cameras, smart appliances,
router, NAS, printer, SSH/FTP/HTTP/Telnet) while staying manageable on an
8 GB honeypot host. Adding more is straightforward (see `docs/05_Containers.md`
§5.6).

## Q7. How is the dataset labeled? Is it ground truth?
**A.** Two layers: (1) per-container regex heuristic emits
`attack_type_hint`; (2) Logstash maps that to `attack_class`. Because every
attack is generated by a script with known intent (every curl in
`03_webattacks.sh` is a deliberate attack), the labels are effectively ground
truth for the lab. In a real Internet deployment you'd need human review or
a semi-supervised approach.

## Q8. Why not use deep learning?
**A.** For tabular features with clean signatures, tree-based ensembles
match or beat deep learning at a fraction of the compute cost. Deep learning
becomes interesting for the raw payload text — see Future Work.

## Q9. How long does training take?
**A.** On the reference hardware (8 cores, 16 GB) with a ~2k-event dataset,
the full `03_train.py` run takes 2–4 minutes. SVM is the slowest; tree
models are near-instant.

## Q10. What accuracy do you achieve?
**A.** On the lab dataset, Random Forest and XGBoost typically reach
>0.98 macro-F1 because honeypot payloads contain very clean signatures (SQL
keywords, shell metacharacters). Real Internet noise would lower this
significantly.

## Q11. Won't the model overfit to the lab's payloads?
**A.** Possibly — that's why Future Work includes adversarial robustness
testing. The current model is a *proof of concept*; a production model
would be retrained on diverse real-world data and tested against evasion
techniques.

## Q12. Is the GeoIP enrichment useful in the lab?
**A.** No — all lab IPs are private (`192.168.1.30`), so GeoIP returns
nothing. We include it for forward compatibility: when the same pipeline is
pointed at Internet-facing honeypots, the geo map populates automatically.

## Q13. What happens if an attacker compromises a container?
**A.** Containers run as non-root, with default Docker seccomp/capabilities
profiles, on an air-gapped network. Even if an attacker escapes the Flask
app to a shell inside the container, they have no path to the host or to
other containers. For added safety we could add a read-only rootfs, an
AppArmor profile, and Falco for runtime detection (Future Work).

## Q14. Why do the SSH/FTP/Telnet honeypots accept weak passwords?
**A.** To capture post-authentication behavior. Real attackers try huge
wordlists; accepting any password for weak usernames lets us observe the
commands they run after logging in, which is the most valuable data for ML.

## Q15. Can I reuse this for my own project?
**A.** Yes — MIT-licensed. Cite as described in the master README §21.
Reusing the intentionally vulnerable containers in production is not
recommended.
